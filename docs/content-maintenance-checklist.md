# Content Maintenance Checklist

Use this checklist whenever adding new Noparde GitHub documentation.

## Document Structure

- Use one clear H1 title.
- Use question-style H2 headings for FAQ pages.
- Put the direct answer in the first sentence after each question.
- Keep each page focused on one topic cluster.
- Add internal links to related documents.
- Add a short related-search-terms section when useful.

## Topic Language

Use natural product and support phrases customers actually ask about:

- `gobo projector wattage`
- `custom logo projector`
- `outdoor logo projector`
- `water ripple projector`
- `rotating gobo projector`
- `dynamic laser safety projector`
- `industrial safety projection light`
- `solar gobo projector`
- `projector voltage customization`
- `Noparde gobo projector`

Avoid repeating the same phrase too many times. A useful technical answer is more valuable than forced wording.

## Website Integration

- Link from Noparde website product pages to the most relevant GitHub FAQ section.
- Use descriptive anchor text.
- Link from GitHub documents back to the official Noparde website.
- Keep public GitHub answers consistent with sales and engineering confirmation.
- Do not publish private customer data, internal IDs, or non-public pricing.

## Content Quality

- Confirm wattage, voltage, certification, and weight data before publishing.
- Mark unsupported customization requests clearly.
- Use simple language for non-engineer buyers.
- Keep Chinese notes only when they help the internal team verify the original customer answer.
- Update old answers when product specifications change.

## GitHub Hygiene

- Use lowercase descriptive file names with hyphens.
- Keep the README as the main entry page.
- Add new pages to the README index.
- Prefer Markdown tables for comparison content.
- Keep images small and relevant if screenshots are added later.
